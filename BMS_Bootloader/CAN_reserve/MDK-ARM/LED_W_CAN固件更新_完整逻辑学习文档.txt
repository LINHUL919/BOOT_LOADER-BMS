========================================================================
一、系统角色说明
========================================================================

本工程是 BMS（电池管理系统）整体方案中的 "CAN 固件接收端"，
运行在 STM32F407 上，起始地址 0x08008000（前 32KB 留给 Bootloader）。

核心功能一句话概括：
  通过 CAN2 总线从上位机接收新固件 → CRC32 硬件校验 →
  写入 W25Q128 外部 SPI Flash → 在 W24C02 EEPROM 置更新标志 →
  复位后由 Bootloader 将固件搬入内部 Flash 完成升级。

系统角色示意：

  +-----------+    CAN 总线     +-------------------+
  |  上位机   | ═══════════════>| STM32F407 (本工程) |
  | (固件发送)|                 | 地址 0x08008000   |
  +-----------+                 +---------+---------+
                                          |
                            SPI1          |          I2C1
                       +----------+       |     +----------+
                       | W25Q128  |<------+---->| W24C02   |
                       | SPI Flash|             | EEPROM   |
                       +----------+             +----------+

工作模式：上电即进入固件接收状态机，无用户交互，全自动流转。

      ├── MX_I2C1_Init()                        [i2c.c]
      │   ├── hi2c1: 100kHz/7bit
      │   └── HAL_I2C_Init() → HAL_I2C_MspInit()
      │       └── PB6/PB7 AF4开漏, I2C1时钟
      ├── MX_SPI1_Init()                        [spi.c]
      │   ├── hspi1: Master/8bit/Mode0/18MHz
      │   └── HAL_SPI_Init() → HAL_SPI_MspInit()
      │       └── PB3/PB4/PB5 AF5, SPI1时钟
      │
      ├── App_Update()                          [app_update.c]
      │   ├── CAN2_Init()                      [Int_can2.c]
      │   │   ├── 过滤器 Bank14 全通(掩码=0x0000)
      │   │   └── HAL_CAN_Start(&hcan2)
      │   ├── printf("CAN Tool Start\r\n")
      │   └── update_state = UPDATE_SEND_CMD
      │
      └── while(1): App_Update_Work()           [app_update.c]
          │
          ├── [UPDATE_SEND_CMD] → do_send_cmd()
          │   ├── memset(buffer, 0)
          │   ├── CAN2_SendData(0, "sss", 3)   [Int_can2.c]
          │   │   ├── 等待TxMailbox空闲
          │   │   └── HAL_CAN_AddTxMessage(StdId=0, 数据帧)
          │   └── state → UPDATE_RECEIVE_DATA
          │
          ├── [UPDATE_RECEIVE_DATA] → do_receive_data()
          │   ├── CAN2_ReceiveData(buf, &cnt)   [Int_can2.c]
          │   │   ├── HAL_CAN_GetRxFifoFillLevel(FIFO0)
          │   │   └── HAL_CAN_GetRxMessage() × N
          │   ├── 逐帧 memcpy 到 app_data_buffer
          │   ├── 溢出检查(>16KB+4)
          │   └── 超时3000ms → state → UPDATE_CRC_CHECK
          │
          ├── [UPDATE_CRC_CHECK] → do_crc_check()
          │   ├── 取末4字节(小端) 为 received_crc
          │   ├── App_crc_cal(data, firmware_len)
          │   │   ├── __HAL_CRC_DR_RESET(&hcrc)
          │   │   └── HAL_CRC_Calculate(按4字节对齐)
          │   ├── 校验通过 → state → UPDATE_WRITE_STORAGE
          │   └── 校验失败 → state → UPDATE_END
          │
          ├── [UPDATE_WRITE_STORAGE] → do_write_storage()
          │   ├── W25Q128_EraseSector(0,0)      [int_w25q128.c]
          │   │   ├── WaitBusy → WriteEnable(0x06)
          │   │   └── SectorErase(0x20) + 24位地址
          │   ├── 循环擦除数据区扇区(按固件大小)
          │   ├── W25Q128_WriteData(0,0,0,0, metadata, 8)
          │   │   ├── WaitBusy → WriteEnable(0x06)
          │   │   └── PageProgram(0x02) + 24位地址
          │   ├── 循环写固件(每次256B, 地址转换)

          │   │   ├── 按8B页对齐自动分段

          │   └── state → UPDATE_END

          └── [UPDATE_END]

                  = 36000000 / (36 * (1+2+2))
                  = 36000000 / 180

  4) 触发操作:
     - 上电/复位后自动开始
     - 无需手动触发, 状态机自动发送 "sss" 请求
     - 上位机收到 "sss" 后按约定协议发送固件帧

────────────────────────────────────────────────────────────────────────
9.1  时钟树与分频关系
────────────────────────────────────────────────────────────────────────

  HSE (25MHz 外部晶振)
    │
    v
  PLL: /M=25 → 1MHz → ×N=144 → 144MHz → /P=2
    │
    v
  SYSCLK = 72 MHz
    │
    ├── AHB Prescaler = /1 → HCLK = 72 MHz
    │   ├── CRC 外设时钟 = 72 MHz
    │   └── GPIO 时钟    = 72 MHz
    │
    ├── APB1 Prescaler = /2 → PCLK1 = 36 MHz
    │   ├── CAN1/CAN2 时钟    = 36 MHz
    │   ├── I2C1 时钟          = 36 MHz
    │   └── Timer2~7,12~14 时钟 = 72 MHz (×2)
    │
    └── APB2 Prescaler = /1 → PCLK2 = 72 MHz
        ├── USART1 时钟 = 72 MHz
        ├── SPI1 时钟   = 72 MHz (→ /4 = 18 MHz SCK)
        └── Timer1,8~11 时钟 = 72 MHz

  Flash等待周期: FLASH_LATENCY_2 (2 WS, 适合 60~90 MHz)

  关键计算:
  - CAN2波特率 = PCLK1 / (Prescaler × (1+BS1+BS2))
               = 36MHz / (36 × 5) = 200 kbps
  - SPI1 SCK   = PCLK2 / Prescaler = 72MHz / 4 = 18 MHz
  - USART1     = PCLK2基础, 115200 baud
  - I2C1       = 100 kHz Standard Mode (由PCLK1=36MHz推导)

────────────────────────────────────────────────────────────────────────
9.2  DMA 通道分配
────────────────────────────────────────────────────────────────────────

  本工程未使用 DMA。所有数据传输均为阻塞/轮询方式:
  - SPI1: HAL_SPI_Transmit / HAL_SPI_Receive (阻塞, 100ms超时)
  - I2C1: HAL_I2C_Mem_Write / HAL_I2C_Mem_Read (阻塞, 100ms超时)
  - CAN2: 轮询 FIFO0 (非中断非DMA)
  - USART1: HAL_UART_Transmit (阻塞, 用于printf)

────────────────────────────────────────────────────────────────────────
9.3  中断优先级分组与抢占关系
────────────────────────────────────────────────────────────────────────

  优先级分组: HAL默认 NVIC_PRIORITYGROUP_4 (4位抢占, 0位子优先级)

  ┌─────────────────────┬──────────┬───────────┬──────────────────────┐
  │ 中断源               │ 抢占优先 │ 子优先级  │ 说明                 │
  ├─────────────────────┼──────────┼───────────┼──────────────────────┤
  │ SysTick             │ 15(默认) │ 0         │ 系统节拍1ms          │
  │ USART1_IRQn         │ 5        │ 0         │ 串口中断             │
  │ NMI/HardFault/...   │ 固定     │ --        │ 系统异常死循环       │
  └─────────────────────┴──────────┴───────────┴──────────────────────┘

  注意:
  - CAN2 未开启中断（未调用 HAL_CAN_ActivateNotification）
  - 仅 SysTick 和 USART1 在运行时产生中断
  - USART1 抢占优先级5可以打断 SysTick(优先级15)

────────────────────────────────────────────────────────────────────────
9.4  外设初始化顺序依赖
────────────────────────────────────────────────────────────────────────

  必须遵从的顺序：
  1. HAL_Init()       → SysTick + NVIC 分组 (其他外设依赖)
  2. SystemClock_Config → PLL 启动 (所有外设时钟的源)
  3. MX_GPIO_Init     → 先配 GPIO 时钟 (后续 MSP 回调需要)
  4. MX_CAN2_Init     → CAN2 底层配置
     注意: CAN2 依赖 CAN1 时钟! (MspInit 里必须同时使能)
           __HAL_RCC_CAN2_CLK_ENABLE() + __HAL_RCC_CAN1_CLK_ENABLE()
  5. MX_USART1_UART_Init → printf 可用
  6. MX_CRC_Init      → CRC 外设 (校验前必须已初始化)
  7. MX_I2C1_Init     → I2C 通道
  8. MX_SPI1_Init     → SPI 通道
  9. App_Update()     → CAN2_Init(过滤器+Start) 必须在 MX_CAN2_Init 之后

  关键依赖: CAN2_Init() 中 HAL_CAN_Start() 必须在
            HAL_CAN_ConfigFilter() 之后、首次收发之前调用！

────────────────────────────────────────────────────────────────────────
9.5  引脚复用冲突检查
────────────────────────────────────────────────────────────────────────

  当前工程引脚分配总览 (Port B 密集使用):
  ┌──────┬──────────────┬───────────────────────────────────────────┐
  │ PB3  │ SPI1_SCK AF5 │ 无冲突 (默认 JTDO/SWO 被复用覆盖)       │
  │ PB4  │ SPI1_MISO AF5│ 无冲突 (默认 NJTRST 被复用覆盖)         │
  │ PB5  │ SPI1_MOSI AF5│ 无冲突                                   │
  │ PB6  │ I2C1_SCL AF4 │ 无冲突                                   │
  │ PB7  │ I2C1_SDA AF4 │ 无冲突                                   │
  │ PB12 │ CAN2_RX AF9  │ 无冲突                                   │
  │ PB13 │ CAN2_TX AF9  │ 无冲突                                   │
  └──────┴──────────────┴───────────────────────────────────────────┘

  *** 注意: PB3/PB4 默认功能是 JTAG 的 JTDO 和 NJTRST。
      使用 SPI1 复用后, JTAG 调试可能受影响。
      建议使用 SWD 模式调试（仅需 PA13/PA14），避免冲突。***

  无引脚复用冲突（前提: 使用 SWD 而非 JTAG）。

────────────────────────────────────────────────────────────────────────
9.6  Flash/EEPROM 擦写注意事项
────────────────────────────────────────────────────────────────────────

  W25Q128 SPI Flash:
  - 擦除最小单位: 4KB 扇区 (命令 0x20)
  - 编程最小单位: 1B, 最大单次 256B (页边界 Page Program)
  - 擦除前数据为 0xFF, 只能从 1→0, 不能直接 0→1
  - 擦除典型耗时: 45ms~400ms, 必须 WaitBusy
  - 编程典型耗时: 0.7ms~3ms, 必须 WriteEnable
  - 当前驱动不处理跨页写入! WriteData 一次调用不能跨256B页边界
    (app_update.c 中 do_write_storage 手动分256B chunk 处理)
  - SPI 时钟 18MHz 在 W25Q128 支持范围内 (最高约 80MHz)

  W24C02 I2C EEPROM:
  - 总容量: 256 字节 (2Kbit)
  - 页大小: 8 字节
  - 写入后必须等待 5ms (tWR, Write Cycle Time)
  - WriteBytes 已实现分页对齐 + 5ms延时
  - I2C 寻址 0xA0 (7位地址 0x50, 最高位为 R/W)
  - 无磨损均衡: 写同一地址超过100万次可能损坏

────────────────────────────────────────────────────────────────────────
9.7  通信时序要求
────────────────────────────────────────────────────────────────────────

  CAN 通信协议 (与上位机约定):
  ┌────────────────────────────────────────────────────────────────────┐
  │ 阶段1: 本端发送请求                                               │
  │   StdId=0, DLC=3, Data="sss"                                     │
  │                                                                    │
  │ 阶段2: 上位机发送固件                                             │
  │   每帧: StdId=任意(过滤器全通), DLC=1~8, Data=固件字节           │
  │   节奏: 每发完256字节, 上位机延时100ms                            │
  │                                                                    │
  │ 阶段3: 上位机发送CRC                                              │
  │   固件全部发完后, 上位机延时2100ms                                │
  │   然后发送4字节CRC32(小端序, 可能需要1帧DLC=4)                   │
  │                                                                    │
  │ 阶段4: 本端超时判定                                               │
  │   最后一帧(CRC)收到后, 3000ms 无新帧 → 认为传输结束             │
  └────────────────────────────────────────────────────────────────────┘

  时序约束:
  - 超时阈值 3000ms > 上位机最大间隔 2100ms, 有 900ms 裕量
  - 上位机必须在发送CRC后不再发帧, 否则本端无法判定结束
  - CAN 帧长度可变(1~8字节), 本端按 DLC 逐帧拼接

────────────────────────────────────────────────────────────────────────
9.8  低功耗相关
────────────────────────────────────────────────────────────────────────

  本工程未涉及低功耗设计:
  - 未使用 STOP / STANDBY / SLEEP 模式
  - 电压调节器保持 SCALE1 (最高性能)
  - 上电后全速运行直到固件写入完成后复位

十、改动总结

本文档为 "只读分析" 类型, 未对工程做任何代码改动。
以下为分析过程中的核心发现与建议:

────────────────────────────────────────────────────────────────────────
10.1  核心发现
────────────────────────────────────────────────────────────────────────

  [发现1] CAN2 实际波特率为 200kbps (已确认与上位机一致, 无冲突)
    根据: Prescaler=36, BS1=2TQ, BS2=2TQ, APB1=36MHz
    计算: 36MHz / (36 × (1+2+2)) = 200kbps
    上位机: CubeMX截图确认 Baud Rate=200000 bit/s, 参数完全一致
    结论: 两端波特率匹配, 无需修改

  [发现2] 本工程 APP 仅分配 16KB Flash (0x08008000~0x0800BFFF)
    16KB 同时是接收缓冲区上限 (APP_DATA_MAX_LEN = 16KB+4)
    适用于较小的应用固件; 若未来 APP 增长需扩展 sct 和缓冲区

  [发现3] PB3/PB4 默认为 JTAG 引脚, 被复用为 SPI1
    调试时必须使用 SWD 模式 (PA13/PA14), 不能使用 JTAG
    Keil Debug 设置需确认已选择 SW 而非 JTAG

  [发现4] CAN2 接收采用轮询模式, 非中断
    主循环每次 App_Update_Work() 调用检查 FIFO0
    优点: 简单可靠, 无需处理中断竞态
    缺点: 主循环其他操作(如 Flash 写入)期间可能丢帧
    当前可接受: 写入操作在收完全部数据之后才执行

  [发现5] CRC32 使用硬件 CRC 外设, 但有字节对齐处理
    App_crc_cal() 将长度向上对齐到4字节再传给 HAL_CRC_Calculate
    若固件长度不是4的倍数, 末尾会有1~3字节填充0参与计算
    上位机的 CRC 算法必须使用相同的对齐策略

────────────────────────────────────────────────────────────────────────
10.2  配置冲突 / 潜在问题
────────────────────────────────────────────────────────────────────────

  [已排除] CAN 波特率 — 无冲突
    严重程度: ☆☆☆☆☆ (已排除)
    can.c Prescaler=36 → 200kbps
    上位机 CubeMX 截图确认: Prescaler=36, BS1=2, BS2=2 → 200kbps
    两端完全一致, 此项不再是问题

  [潜在问题1] I2C 总线无外接上拉电阻提示
    i2c.c 配置 Pull=GPIO_NOPULL (MCU 内部不上拉)
    I2C 开漏模式必须有外部上拉电阻 (推荐 4.7KΩ)
    若 PCB 未焊接上拉电阻, I2C 通信会失败

  [潜在问题2] CRC 字节序与对齐
    发送端(上位机) CRC 是小端序: LSB 在前
    接收端按 (uint32_t*) 强转后直接传给 HAL_CRC_Calculate
    若 app_data_buffer 地址不是 4 字节对齐, 在部分编译器下
    可能触发 HardFault (但 Keil ARM CC 对 static 数组通常
    自动 8 字节对齐, 实际风险低)

  [潜在问题3] W25Q128 写入不检查返回值
    当前所有 W25Q128 操作无错误处理
    若 SPI 通信异常, 数据静默损坏
    建议: 写入后读回验证关键数据
