========================================
一、系统角色说明
========================================

本模块是 BMS(电池管理系统) + Bootloader 方案中的【上位机(Host)端固件】。

核心职责：
  1. 存储待更新的应用程序固件（存放于自身Flash的 0x08008000 区域）
  2. 通过 CAN 总线监听下位机(目标板)发来的更新请求("sss")
  3. 收到请求后，将固件数据按 8字节/帧 分块通过 CAN 发送给下位机
  4. 发送完毕后计算并发送 CRC32 校验值，供下位机验证数据完整性

附加功能(当前被注释、备用)：
  - 通过 UART 接收 PC 端发来的应用固件并写入自身 Flash（App_bootloader模块）
  - 该模式用于"先把新固件灌入上位机，再由上位机通过CAN转发给下位机"

系统拓扑：

  +-----------+    UART     +--------------+    CAN Bus    +--------------+
  |   PC端    | ----------> |  上位机(本机) | -----------> |  下位机/目标 |
  | 串口工具  |  115200bps  | STM32F103C8  |   250kbps    | STM32+BMS    |
  +-----------+             +--------------+               +--------------+
                            Flash布局:                      bootloader区
                            0x08000000 本机程序              接收CAN数据
                            0x08008000 待发送的APP           写入Flash
                                                            CRC校验后跳转

二、内存/Flash/外设地址划分图
========================================

  Flash 地址空间 (64KB = 0x10000)
  +-----------------------------------------------------+
  | 0x08000000 |  本机上位机程序 (Bootloader+上位机逻辑) | 32KB
  |            |  链接脚本: 0x08000000 ~ 0x0800FFFF      |
  |            |  包含中断向量表、代码、只读数据          |
  +------------+-----------------------------------------+
  | 0x08008000 |  应用程序存储区 (待发送给下位机的固件)   | 32KB
  |            |  APP_START_ADDRESS = 0x08008000          |
  |            |  APP_END_ADDRESS   = 0x08010000          |
  |            |  每页1KB, 共32页 (APP_PAGE_COUNT=32)     |
  +------------+-----------------------------------------+
  | 0x08010000 |  Flash结束                               |
  +-----------------------------------------------------+

  SRAM 地址空间 (20KB = 0x5000)
  +-----------------------------------------------------+
  | 0x20000000 |  RW数据 + ZI数据 + 堆栈                 | 20KB
  |            |  链接脚本: 0x20000000 ~ 0x20004FFF      |
  +-----------------------------------------------------+

  外设基地址 (APB1/APB2)
  +-----------------------------------------------------+
  | 0x40000000 |  APB1 总线: CAN1 (bxCAN)               |
  | 0x40006400 |  CAN1 基地址                            |
  +------------+-----------------------------------------+
  | 0x40010000 |  APB2 总线                              |
  | 0x40013800 |  USART1 基地址                          |
  +------------+-----------------------------------------+
  | 0x40023000 |  AHB 总线: CRC 模块                     |
  +-----------------------------------------------------+

  链接脚本 (test_CAN_HAL.sct):
    LR_IROM1  0x08000000  0x00010000   ; 加载域: 64KB Flash全部
    ER_IROM1  0x08000000  0x00010000   ; 执行域: Flash
    RW_IRAM1  0x20000000  0x00005000   ; 数据域: 20KB SRAM

  [已确认无冲突]
  虽然 .sct 文件中写的是 0x10000(64KB)，但 Keil Target 设置中
  IROM1 = 0x08000000, Size = 0x8000(32KB)，Keil 使用 Target Dialog
  的内存布局覆盖 .sct，实际程序空间为 0x08000000~0x08007FFF，
  与 0x08008000 应用存储区不冲突。

4.2 并行中断入口关系（箭头简图）
---------------------------------------------------------------------------

  SysTick (1ms)
    SysTick_Handler()  ->  HAL_IncTick()  ->  更新uwTick全局计数器

  USART1中断 (仅App_bootloader模式启用)
    USART1_IRQHandler()
      -> HAL_UART_IRQHandler(&huart1)

---------------------------------------------------------------------------

  +-------------------+
  |    上电/复位      |
  +--------+----------+
           |
           v
  +-------------------+
  |  HAL_Init()       |
  |  SystemClock 72MHz|
  |  GPIO/CAN/UART/CRC|
  +--------+----------+
           |
           v
  +-------------------+
  | App_update_init() |

  | - 默认len=1024    |
  +--------+----------+
           |
           v
  +=============================+
  | 主循环: App_update_work()   |
  +=============================+
           |
           v
  +----------------------------+       +----------------------------+
  | 状态: WAIT_FOR_REQUEST     |       | 状态: SENDING_APP          |

                                               +------------------+

                                               | 回到WAIT状态     |
                                               | (可重复发送)     |
                                               +------------------+

  发送数据帧格式:
    CAN ID = 1 (app_update_request_ID)
    DLC = 1~8字节 (最后一帧可能<8)

  CRC帧格式:
    CAN ID = 1, DLC = 4字节
    data[0]=CRC[31:24] data[1]=CRC[23:16]
    data[2]=CRC[15:8]  data[3]=CRC[7:0]  (大端序)

      -> 越界检查(>0x08010000则return)
      -> HAL_FLASH_Unlock()
      -> for i+=2: 半字写入(奇数字节补0xFF)
      -> flash_write_offset偶数对齐
      -> HAL_FLASH_Lock()
      -> 清空buffer -> 重启中断接收

9.1 时钟树与分频关系
---------------------------------------------------------------------------
  HSE(8MHz) -> PLL x9 -> SYSCLK = 72MHz
                |
                +-> AHB  /1 = 72MHz  -> Flash, SRAM, DMA, CRC
                +-> APB1 /2 = 36MHz  -> CAN1, TIM2-4, USART2-3
                +-> APB2 /1 = 72MHz  -> USART1, GPIO, AFIO, ADC
                +-> SysTick  = 72MHz / 8 = 9MHz (或72MHz直接)

  Flash等待周期: FLASH_LATENCY_2 (72MHz需要2个等待周期)

9.2 中断优先级分组与抢占关系
---------------------------------------------------------------------------
  STM32F1 HAL默认: NVIC_PRIORITYGROUP_4 (4位抢占, 0位子优先级)
  但本工程未显式调用 HAL_NVIC_SetPriorityGroup()，
  HAL_Init()默认设置为 NVIC_PRIORITYGROUP_4。

  当前中断配置:
    +-------------------+--------+---------+--------+
    | 中断源            | 抢占级 | 子优先级| 说明   |
    +-------------------+--------+---------+--------+
    | SysTick           |  15    |   0     | 最低   |
    | USART1_IRQn       |   0    |   0     | 最高   |
    +-------------------+--------+---------+--------+

  CAN未配置中断(纯轮询), 不占中断资源。
  SysTick默认优先级最低(15), UART最高(0), 不会互相抢占问题。

9.3 外设初始化顺序依赖
---------------------------------------------------------------------------
  必须遵循以下顺序:
    1. HAL_Init()        -- 初始化Flash接口和SysTick
    2. SystemClock_Config() -- PLL/时钟树必须先配好
    3. MX_GPIO_Init()    -- GPIO时钟必须在外设引脚配置之前使能
    4. MX_CAN_Init()     -- CAN初始化(MspInit中配GPIO)
    5. MX_USART1_UART_Init() -- UART初始化
    6. MX_CRC_Init()     -- CRC初始化(无依赖,但放最后)
    7. CAN_Init()        -- 过滤器+启动CAN(必须在MX_CAN_Init之后)

  注意: CAN_Init()在App_update_init()中调用, 必须在HAL_CAN_Init()之后。
  如果调换顺序会导致CAN外设未初始化就配置过滤器 -> HardFault。

9.4 引脚复用冲突检查
---------------------------------------------------------------------------
  PA9  = USART1_TX  (无冲突)
  PA10 = USART1_RX  (无冲突)
  PA11 = CAN1_RX    (与USB_DM共用! F103的PA11同时是USB D-)
  PA12 = CAN1_TX    (与USB_DP共用! F103的PA12同时是USB D+)

  !! 注意 !! PA11/PA12 在 STM32F103 中 CAN 和 USB 共用引脚,
  不能同时使用CAN和USB。本工程仅用CAN,无冲突。
  若需同时使用USB, 需将CAN重映射到PB8(RX)/PB9(TX)。

  PA13 = SWDIO (调试用, 保留)
  PA14 = SWCLK (调试用, 保留)
  PA15/PB3/PB4 = JTAG已释放, 可用作GPIO(当前未使用)

9.5 Flash/EEPROM擦写注意事项
---------------------------------------------------------------------------
  STM32F103C8T6 Flash特点:
    - 总容量: 64KB (部分实际有128KB但不保证)
    - 页大小: 1KB (0x400)
    - 擦除: 只能按页擦除, 不能按字节
    - 写入: 只能半字(16位)写入, 地址必须2字节对齐
    - 写入前必须擦除(1->0), 不能0->1
    - 操作前 HAL_FLASH_Unlock(), 操作后 HAL_FLASH_Lock()
    - 擦写期间 Flash 不可读(代码不能从Flash执行)
      -> 如果在中断中擦写,主循环不能访问Flash
      -> 本工程在中断回调中写Flash, 注意此限制

  本工程应用区:
    起始: 0x08008000 (第32页)
    结束: 0x08010000 (第64页)
    共32页, 32KB空间

  奇数字节处理:
    当接收字节数为奇数时, 最后一字节高位补0xFF:
    data = buf[i] | 0xFF00
    这意味着奇数长度的固件最后一字节后会多一个0xFF

9.6 CAN通信时序要求
---------------------------------------------------------------------------
  - CAN发送等待所有3个邮箱空闲: while(FreeLevel < 3)
    这保证了帧按FIFO顺序到达, 但降低了吞吐量
  - 每256字节延时100ms: 给接收端处理+写Flash时间
  - 发送完毕延时2100ms: 确保最后的数据帧全部到达后再发CRC
  - CRC帧与数据帧使用相同的CAN ID(=1), 接收端需按时序区分

  !! 潜在问题 !!
  CRC帧和数据帧ID相同, 接收端区分它们的唯一方式是"超时":
  数据帧间隔 < 2.1秒, CRC帧在数据帧结束2.1秒后到达。
  如果CAN总线拥塞导致延迟, 可能误判。

9.7 AFIO与JTAG配置
---------------------------------------------------------------------------
  __HAL_AFIO_REMAP_SWJ_NOJTAG(): 禁用JTAG, 仅保留SWD
  效果: PA15/PB3/PB4 可用作普通GPIO
  注意: 此操作在HAL_MspInit()中执行, 即HAL_Init()期间
  如果代码跑飞导致无法SWD连接, 需要按住RESET再连接
